## 🔴 Critical Issues Found & Attempted Fixes

### Issue #1: Infinite Loop on "China" Questions (FIXED ✓)

**Problem**:  
When user asks "Who is the president of China?", Qwen 0.5B model enters infinite loop generating Chinese tokens. The model generates thousands of tokens like: Chinese characters + "|im_end|" + more Chinese + role markers.

**Root Cause**:  
- Qwen is Chinese-English bilingual model
- When asked about China, it switches to Chinese mode
- No wall-clock timeout in C++ - only token count limit
- Chinese uses many short tokens, so stays under token limit while generating forever

**Fix Applied**:  
File: `native_android/app/src/main/cpp/ai_chat_jni.cpp`
Added 30-second hard timeout in generation loop:
```cpp
auto start_time = std::chrono::steady_clock::now();
const int MAX_GENERATION_SECONDS = 30;

while (n_gen < maxTokens && !g_state->should_stop) {
    // Check hard timeout
    auto elapsed = std::chrono::duration_cast<std::chrono::seconds>(
        std::chrono::steady_clock::now() - start_time).count();
    if (elapsed > MAX_GENERATION_SECONDS) {
        LOGW("HARD TIMEOUT: Generation stopped after %d seconds at token %d", 
             MAX_GENERATION_SECONDS, n_gen);
        break;
    }
    // ... rest of loop
}
```

**Status**: ✓ FIXED - China questions now timeout properly and return partial response

### Issue #2: TTS/LLM Sync - "TTS Starts After Full Generation" (NOT FIXED ✗)

**Problem**:  
User reports TTS should start when first sentence is ready (streaming), not wait for full LLM generation.

**Current Behavior**:  
1. LLM generates full response
2. Then TTS starts speaking
3. User experiences delay before hearing voice

**Expected Behavior**:  
1. LLM starts generating
2. First sentence ready → TTS starts immediately (low latency)
3. TTS streams as more text arrives
4. When TTS done → transition to LISTENING

**Attempts Made**:

**Attempt 1**: Modified `startStreamingTTS()` to accept `initialText` parameter
- Called when `hasCompleteSentence()` returns true during token streaming
- Result: TTS starts but doesn't properly stream subsequent text

**Attempt 2**: Removed streaming TTS entirely, made it blocking
- Wait for full text → speak all → then listen
- Result: Too slow, user complains about delay

**Attempt 3**: Reverted to hybrid approach
- Start TTS when first sentence ready
- Continue speaking as text arrives via `ttsTextBuffer`
- Result: Partially works but still has timing issues

**Current Code State**:  
File: `VoicePipelineManager.kt`
```kotlin
response.token != null && !response.isComplete -> {
    // STREAMING: Collect tokens, update UI, start TTS when first sentence ready
    tokenCount++
    transitionToState(PipelineState.SPEAKING)
    fullResponseText.append(response.token)
    
    // Update UI in real time
    withContext(Dispatchers.Main) {
        onResponseUpdate?.invoke(fullResponseText.toString(), false)
    }
    
    // Start TTS as soon as we have a complete sentence
    val currentText = truncateAtRoleLeakage(fullResponseText.toString())
    if (!isStreamingTTSActive && hasCompleteSentence(currentText)) {
        Log.i(TAG, "First sentence ready, starting streaming TTS")
        startStreamingTTS(currentText)
    }
}
```

**Why It's Not Working**:  
- `startStreamingTTS()` runs in a coroutine that polls for new text
- The `ttsTextBuffer` approach has race conditions
- When `isLLMResponseComplete` is set, TTS may have already finished or not
- The `finally` block in `startStreamingTTS()` transitions to LISTENING even if LLM still running

**Status**: ✗ NOT FIXED - Needs redesign

## 📋 Current Pipeline Flow (As of Commit 38a9a79)

```
User speaks
    ↓
VAD detects speech
    ↓
STT transcribes → "Who is PM of Nepal?"
    ↓
State: THINKING
    ↓
LLM starts generating tokens
    ↓
UI updates in real-time (streaming text)
    ↓
First sentence ready? → Start TTS (if not already started)
    ↓
TTS speaks sentence-by-sentence (QUEUE_ADD mode)
    ↓
LLM completes (EOS or max tokens)
    ↓
finalBubbleText = fullResponseText
    ↓
isLLMResponseComplete = true
    ↓
TTS speaks remaining text
    ↓
TTS finally block:
    - isStreamingTTSActive = false
    - transitionToListening()
    ↓
State: LISTENING
    ↓
Ready for next question
```

**Known Issues in This Flow**:  
1. TTS may not start until full generation if `hasCompleteSentence()` check fails
2. `isLLMResponseComplete` flag may cause race conditions
3. `ttsTextBuffer` may get out of sync with `fullResponseText`
