## Problems Encountered & Solutions

### Problem 1: LLM Stopping Mid-Sentence at `<|im_end|>`

**Symptom**: When asking questions like "Who is the prime minister of Nepal?", the LLM would generate:
```
"The current ruler of Nepal is the King of Nepal, Maha Bhakta Indra Prasad Acharya, who was previously known as King Prithvi Narayan Shah. He ascended to the throne in 2008 and has been ruling since then. However, it's worth noting that there"
```
The response would stop mid-sentence and the app would go to listening mode.

**Root Cause 1** (Native C++ layer): `ai_chat_jni.cpp` had stop logic at lines 714 and 757-762:
```cpp
// Token-level stop
if (new_token == token_im_end || new_token == token_im_start) {
    break;  // STOPPED generation at im_end!
}

// String-level stop
if (display_str.find("<|im_end|>") != std::string::npos) {
    break;  // STOPPED generation at im_end!
}
```

**Root Cause 2** (Kotlin layer): `VoicePipelineManager.kt` had `truncateAtRoleLeakage()` function that truncated ALL text after `<|im_end|>` marker.

**Solution Applied**:
1. **Native layer** (`ai_chat_jni.cpp`):
   - Removed `token_im_end` from token-level stop condition
   - Removed `<|im_end|>` and `|im_end|>` from string-level stop sequences
   - Now only stops at `im_start` (actual role transition)

2. **Kotlin layer** (`VoicePipelineManager.kt`):
   - Created `removeImEndTokenOnly()` function that removes just the marker but keeps all text after it
   - Updated all display/TTS calls to use this instead of `truncateAtRoleLeakage()`
   - Removed `<|im_end|>` from `stopSequences` in `CompletionRequest`

### Problem 2: TTS Completion Delay

**Symptom**: After TTS finished speaking, the app stayed in "Nilo is speaking..." state for several extra seconds before switching to "Nilo is listening..."

**Root Cause**: `VoicePipelineManager.kt` used a fixed delay calculation:
```kotlin
val waitTimeMs = ((totalTextLength / 12.0) * 1000).toLong().coerceIn(3000L, 25000L)
delay(waitTimeMs)  // Just waits blindly!
```

This calculated ~20 seconds for long responses even if TTS finished in 5 seconds.

**Solution Applied**:
```kotlin
// Poll TTS completion instead of fixed delay
var ttsWaitCount = 0
val maxTtsWaitMs = 30000
while (ttsWaitCount < maxTtsWaitMs) {
    if (kokoroTTS?.isSpeaking() != true) {
        break  // TTS actually finished!
    }
    delay(100)
    ttsWaitCount += 100
}
```

Now polls `isSpeaking()` every 100ms for immediate transition.

### Problem 3: TTS Speaking Text Not in UI Bubble

**Symptom**: TTS would speak hallucinated text after `<|im_end|>` but the UI bubble wouldn't show it.

**Root Cause**: Mismatch between TTS source text and UI display text:
- TTS used `truncateAtRoleLeakage()` - truncated at `<|im_end|>` 
- UI used `removeImEndTokenOnly()` - showed all text

Actually this was the opposite - TTS was speaking MORE than the bubble showed because we changed to use `removeImEndTokenOnly()` for both.

**Key Functions Created**:
```kotlin
// Removes ONLY the marker, keeps all content
private fun removeImEndTokenOnly(text: String): String {
    return text
        .replace("<|im_end|>", "")
        .replace("|im_end|>", "")
        .replace("|im_end|", "")
        .replace("im_end|", "")
}

// OLD - truncates everything after marker (don't use for display)
private fun truncateAtRoleLeakage(text: String): String {
    // Finds <|im_end|> and returns only text before it
}
```

## Key Learnings for Future AI Agents

1. **Native Layer Controls Generation**: The C++ code in `ai_chat_jni.cpp` has ultimate control over when generation stops. If text is being truncated, check there first.

2. **TTS Completion Detection**: Never use fixed delays for TTS completion. Always poll `kokoroTTS?.isSpeaking()` for accurate detection.

3. **Token Markers**: The model generates `<|im_end|>` as an end-of-turn marker, but we want to show ALL generated text to the user (just hide the marker).

4. **Two Stop Layers**: 
   - Token-level: C++ checks `new_token == token_im_end`
   - String-level: C++ checks `display_str.find("<|im_end|>")`
   Both need to be modified for the fix to work.

5. **Build Process**: After modifying C++ code, must run:
   ```bash
   ./gradlew clean
   ./gradlew :app:assembleDebug
   ```
   A simple Java rebuild won't update the native library.

## Known Limitations

1. **Model Hallucination**: The model sometimes generates incorrect facts (e.g., outdated PM names). This is a model limitation, not a code bug.

2. **30-Second Timeout**: The native layer has a hard 30-second timeout for generation. Very long responses may get cut off.

3. **First Query Slow**: Initial model loading takes 5-10 seconds. Subsequent queries are faster.
